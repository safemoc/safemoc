# Lucario Agent Rules（Lucario项目执行规范）

## 1. 职责范围

Agent 可参与：

- Backend（后端）
- Frontend（前端）
- Documentation（项目文档）

但必须遵循严格边界控制，不允许擅自跨模块修改。

---

## 2. 前后端修改边界

### 默认原则

Agent 不得主动修改前端或后端。

只有在任务描述中出现明确指令时，才允许修改对应模块。

允许示例：

- 修改前端登录页面
- 调整用户列表接口
- 修改前后端联调逻辑
- 更新 Dashboard 展示

不允许示例：

用户说：

> 给我优化一下这个功能

Agent 不得自行推断：

- 顺手改接口
- 顺手改UI
- 顺手调整数据库
- 顺手修改状态管理
- 顺手修改前端展示

必须先确认：

```txt
本次需要修改：

[ ] Frontend
[ ] Backend
[ ] Both

请确认范围。
````

未经确认不得跨边界修改。

---

## 3. 文档优先原则（强制）

任何代码变更前：

必须先更新文档，再修改代码。

执行顺序：

1. 分析需求
2. 梳理影响范围确认实现方案
3. 等待确认（如需要）
4. 更新文档
5. 修改代码
6. 验证
7. 更新最终文档

禁止：

代码改完后补文档。

---

## 4. 文档更新要求

每次变更必须同步更新：

可能包括：

* API 文档
* 数据结构
* 数据库设计
* 流程图
* 状态流转
* 权限设计
* README
* 模块说明

若无需更新文档，也必须明确说明：

```txt
本次改动无需更新文档。

原因：
仅内部实现调整，无接口、结构、行为变化。
```

禁止跳过。

---

## 5. 业务逻辑不明确禁止执行

如果需求存在：

* 歧义
* 缺少条件
* 边界不清晰
* 状态不完整
* 权限不明确
* 数据来源不明确
* 异常流程不明确

必须先提问。

禁止：

自行猜测业务。

错误示例：

用户：

> 增加审批功能

禁止直接实现。

必须先确认：

```txt
请确认：

1. 谁可以审批
2. 审批几级
3. 是否允许撤回
4. 审批状态有哪些
5. 审批失败如何处理
6. 是否通知
7. 是否记录日志
8. 是否支持历史查询
```

未确认：

禁止编码。

---

## 6. 风险变更强制确认

若改动可能影响：

### 功能

* 已有业务
* 老接口
* 历史数据
* 页面行为
* 权限逻辑
* 状态流转

### 技术

* 数据库结构
* 公共组件
* 公共方法
* 中间件
* SDK
* 缓存
* 登录流程

必须停止执行并提示：

```txt
检测到该修改可能影响：

1. xxx
2. xxx
3. xxx

潜在风险：

- xxx
- xxx

建议方案：

xxx

是否继续？
```

未确认：

禁止执行。

---

## 7. 高风险操作禁止直接执行

以下行为必须二次确认：

### 数据

* 删除数据
* 数据迁移
* 批量更新
* 清库
* 改表

### 系统

* 删除接口
* 重构核心逻辑
* 权限调整
* 登录流程修改
* 支付流程修改

确认格式：

```txt
检测到高风险操作：

操作：
xxxx

影响：
xxxx

是否确认执行？
```

---

## 8. 不允许隐式行为

禁止：

* 顺手优化
* 顺手重构
* 顺手修 Bug
* 顺手升级依赖
* 顺手调整代码结构
* 顺手修改命名

除非明确要求：

```txt
允许重构
允许优化
允许修复关联问题
```

否则只能修改目标范围。

---

## 9. 执行前输出计划

正式执行前必须输出：

```txt
任务理解：

xxx

影响范围：

- Frontend
- Backend
- Docs

执行步骤：

1.xxx
2.xxx
3.xxx

风险：

xxx

待确认项：

xxx
```

确认后执行。

---

## 10. 执行完成后输出总结

必须输出：

```txt
完成内容：

1.xxx
2.xxx

修改文件：

- xxx
- xxx

文档更新：

- xxx

影响：

无
或：

影响xxx

后续建议：

xxx
```

---

## 核心原则

1. 不猜业务
2. 文档先于代码
3. 不跨边界修改
4. 风险必须提示
5. 不做隐式改动
6. 有疑问先问
7. 有影响先确认
8. 小改动也遵守流程

宁可多问一次，不允许猜测执行。
